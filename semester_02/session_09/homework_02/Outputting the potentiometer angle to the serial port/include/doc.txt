
// Для стандартных тактовых кнопок в проектах на Arduino оптимальной считается
// частота опроса в диапазоне от 20 до 100 Гц (интервал 10–50 мс). Этот
// промежуток позволяет эффективно подавлять дребезг контактов, сохраняя при
// этом мгновенную реакцию на действия пользователя.
///////////////////////////////////////////////////////////////////////////////
// Ключевые рекомендации по частоте
// 1. Низкая (100–200 мс): Максимально допустимая задержка, которую человек еще
// не воспринимает как «торможение» интерфейса.
// 2. Стандартная (20–50 мс): Оптимально для большинства бытовых кнопок. Этого
// времени достаточно, чтобы механические
//    колебания контактов («дребезг») утихли.
// 3. Высокая (5–10 мс): Используется в динамичных приложениях (например,
// игровые контроллеры), где важна минимальная
//    задержка ввода.
///////////////////////////////////////////////////////////////////////////////
// Методы реализации
// 1. Программная задержка (delay): Самый простой способ — поставить паузу после
// считывания. Минус: Блокирует выполнение
//    остального кода.
// 2. Неблокирующий опрос (millis): Рекомендуемый метод. Программа проверяет
// состояние кнопки только тогда, когда прошло
//    нужное количество миллисекунд.
// 3. Прерывания по таймеру: Опрос кнопки строго каждые 1–2 мс внутри
// обработчика прерывания таймера (например, Timer0)
//    позволяет добиться высочайшей стабильности и освободить основной цикл
//    loop.
///////////////////////////////////////////////////////////////////////////////
// Важно: Слишком частый опрос (быстрее 1–2 мс) избыточен и может привести к
// ложным срабатываниям из-за того, что алгоритм «увидит» каждое колебание
// дребезга как отдельное нажатие. Оптимальная частота опроса кнопки для Arduino
// зависит от задачи и особенностей работы кнопки. Важно учитывать, что сигнал
// кнопки при замыкании и размыкании может иметь дребезг контактов, который
// нужно «погасить» (debounce) для однозначного определения состояния кнопки.
// Также нужно избегать помех, которые могут приводить к ложным нажатиям кнопки,
// например при длинных проводах до кнопки или наличии рядом индуктивной
// нагрузки. Рекомендации
// 1. Периодический опрос. Например, каждые 50 мс (20 раз в секунду) —
// достаточно, чтобы «пропустить» дребезг даже от
//    самой плохой кнопки и снизить вероятность поймать внешние «помехи».
// 2. Учёт времени устойчивого состояния сигнала. Рекомендуется выбирать это
// время в пределах 20–30 мс. Нужно дождаться,
//    когда в течение определённого времени состояние кнопки будет стабильным, и
//    только тогда принимать решение.
// 3. Обработка изменения состояния. Клик — это событие, происходящее при
// изменении одного состояния на другое, а не
//    время удержания кнопки. Именно изменение состояния и нужно обрабатывать,
//    чтобы избежать предсказывания реакции пользователя.
///////////////////////////////////////////////////////////////////////////////
// Учёт режима работы кнопки.
// Например, для «высокоскоростных» комбинаций может хватать периода опроса 10
// мс, но для музыкальных целей такого периода, скорее всего, окажется
// недостаточно. Если логика программы реализована в виде циклического или
// однопроходного алгоритма, опрос состояния кнопки можно производить в конце
// или начале каждого нового цикла программы.
///////////////////////////////////////////////////////////////////////////////
// Код
// Для опроса кнопки с определённой частотой можно использовать прерывание по
// таймеру. Если разместить функцию обработки кнопки в прерывании таймера,
// которое будет выполняться с некоторым постоянным интервалом, то опрос кнопки
// может производиться независимо от состояния фоновой программы. Для работы с
// таймерами рекомендуют специальную библиотеку, например, TimerInterrupt.h.
// Важно: для надёжной работы с кнопками метод, отслеживающий состояние кнопки,
// нельзя вызывать более одного раза на один проход loop() — в противном случае
// можно пропускать события кнопки.
///////////////////////////////////////////////////////////////////////////////
// Ошибки
// 1. Ложные срабатывания. Если не обрабатывать дребезг контактов, на каждое
// нажатие кнопка вырабатывает много фронтов
//    сигналов, и программа инвертирует состояние светодиода на каждый фронт.
// 2. Пропускание событий. Если функция, отслеживающая состояние кнопки,
// запускается слишком редко, она не может
//    гарантированно получить данные о её состоянии.
// 3. Неправильная последовательность сигналов. Например, если частота опроса
// установлена 4 мс, контроллер может видеть
//    зажатые выводы в неправильной последовательности.
// Чтобы избежать ошибок, рекомендуется проводить эксперименты и настраивать
// частоту опроса с учётом особенностей кнопки и задачи.

// Способ подключения кнопки к контроллеру. В зависимости от выбранного
// способа, логический уровень, соответствующий состоянию "кнопка не нажата",
// может быть HIGH или LOW.
// 1. С подтягивающим резистором к питанию (Pull-Up).
//    Когда кнопка не нажата, пин контроллера через резистор 10 кОм подключён

//    к GND → LOW. При нажатии кнопка замыкает пин на +5V → HIGH.
